## Task Generation Rules

**CRITICAL**: Tasks MUST be organized by user story to enable independent implementation and testing.

**Tests follow TDD by default**: Unit tests are generated for every user story before implementation tasks. To skip test generation, the user must explicitly pass `skip-tests` in `$ARGUMENTS`.

### Checklist Format (REQUIRED)

Every task MUST strictly follow this format:

```text
- [ ] [TaskID] [P?] [Est:N] [Story?] Description with file path
```

**Format Components**:

1. **Checkbox**: ALWAYS start with `- [ ]` (markdown checkbox)
2. **Task ID**: Sequential number (T001, T002, T003...) in execution order
3. **[P] marker**: Include ONLY if task is parallelizable (different files, no dependencies on incomplete tasks)
4. **[Est:N] marker**: Estimated effort in story points (1/2/3/5/8/13 Fibonacci scale). Include on ALL tasks.
5. **[Story] label**: REQUIRED for user story phase tasks only
   - Format: [US1], [US2], [US3], etc. (maps to user stories from spec.md)
   - Setup phase: NO story label
   - Foundational phase: NO story label  
   - User Story phases: MUST have story label
   - Polish phase: NO story label
6. **Description**: Clear action with exact file path

**Examples**:

- ✅ CORRECT: `- [ ] T001 [Est:2] Create project structure per implementation plan`
- ✅ CORRECT: `- [ ] T005 [P] [Est:3] Implement authentication middleware in src/middleware/auth.py`
- ✅ CORRECT: `- [ ] T012 [P] [Est:3] [US1] Create User model in src/models/user.py`
- ✅ CORRECT: `- [ ] T014 [Est:5] [US1] Implement UserService in src/services/user_service.py`
- ❌ WRONG: `- [ ] Create User model` (missing ID, estimate, and Story label)
- ❌ WRONG: `T001 [Est:3] [US1] Create model` (missing checkbox)
- ❌ WRONG: `- [ ] [US1] Create User model` (missing Task ID and estimate)
- ❌ WRONG: `- [ ] T001 [US1] Create model` (missing estimate and file path)

### Task Organization

1. **From User Stories (spec.md)** - PRIMARY ORGANIZATION:
   - Each user story (P1, P2, P3...) gets its own phase
   - Map all related components to their story:
     - Models needed for that story
     - Services needed for that story
     - Endpoints/UI needed for that story
     - Unit tests for that story (TDD default -- write before implementation)
   - Mark story dependencies (most stories should be independent)

2. **From Contracts**:
   - Map each contract/endpoint → to the user story it serves
   - Each contract → contract test task [P] before implementation in that story's phase (unless skip-tests)

3. **From Data Model**:
   - Map each entity to the user story(ies) that need it
   - If entity serves multiple stories: Put in earliest story or Setup phase
   - Relationships → service layer tasks in appropriate story phase

4. **From Setup/Infrastructure**:
   - Shared infrastructure → Setup phase (Phase 1)
   - Foundational/blocking tasks → Foundational phase (Phase 2)
   - Story-specific setup → within that story's phase

5. **From Migration Strategy** (if plan.md has a Migration Strategy section):
   - Generate migration script task (create the schema migration)
   - Generate rollback migration task (create the reverse migration)
   - Generate migration test task (test migration with production-like data volume)
   - Generate data backfill task (if backfill approach specified)
   - Place migration tasks in Foundational phase (Phase 2) if they block user stories
   - Add dependency markers: user story tasks that depend on the migrated schema must wait for migration tasks

### Phase Structure

- **Phase 1**: Setup (project initialization)
- **Phase 2**: Foundational (blocking prerequisites - MUST complete before user stories)
- **Phase 3+**: User Stories in priority order (P1, P2, P3...)
  - Within each story: Unit Tests (TDD default) → Models → Services → Endpoints → Integration → Run Tests & Report → Fix Failures
  - Each phase should be a complete, independently testable increment
- **Final Phase**: Polish & Cross-Cutting Concerns (run full test suite, report coverage, fix regressions)
